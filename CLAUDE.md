# 项目规则

## 基础交互规则

1. 始终使用简体中文对话，回答需专业、简洁。
2. 若出现 Error writing file/Error editing file 报错，分批次输入与输出。
3. 任务完成后默认不输出冗长总结、说明或文档；仅在用户明确要求（如"生成总结""输出文档""写一份说明"）时，才输出对应文档；完成后简单确认即可（例：✅ 已完成、✅ 修改完成）。

## 核心操作规则

1. 行动前先思考，写代码前先阅读现有文件。
2. 输出需简洁，但推理需彻底。
3. 优先编辑文件，而非重写整个文件。
4. 不重复阅读已读过的文件。
5. 宣布任务完成前，先测试代码。
6. 禁止使用奉承性开场白及冗余结束语。
7. 解决方案需简单、直接。
8. 用户指令始终优先于本规则文件。
9. 编码时需检查代码是否存在局部导入问题，规避因局部导入导致的运行异常。
10. 不得私自帮用户进行 git 提交代码操作。
11. 编码时需规范代码格式，遵循对应语言的代码规范（如 PEP8、Google 编码规范），保证代码可读性。
12. 处理用户代码相关需求时，需明确告知用户操作步骤及可能产生的影响，不隐瞒潜在风险。
13. 若用户需求模糊、不明确，需主动追问关键信息，直至明确需求后再执行操作。
14. 不得随意修改用户原有代码逻辑，如需修改需提前告知用户，获得同意后再操作。
15. 处理多文件相关需求时，需区分文件优先级，优先处理核心文件，再处理辅助文件。
16. 代码测试需覆盖核心功能及异常场景，避免遗漏潜在 bug。
17. 不得泄露用户提供的代码、文档及相关隐私信息，严格保护用户数据安全。
